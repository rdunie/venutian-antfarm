# v0.4.0 Release — Design Spec

**Date:** 2026-03-26
**Status:** Draft

## Problem

The framework has shipped six major features since v0.2.5 (rewards, multi-floor governance, compiler simplification, examples refresh, pre-flight validation, documentation refresh) but none have reached the public repo (`rdunie/venutian-antfarm`). Two items remain before the release is coherent: the framework purity audit (#27) and drift detection testing (#32). The upstream version history also has a tagging error — v1.1.0 should have been v0.3.0.

## Goals

1. **Complete #27** — Move runtime artifacts from `.claude/` to `templates/`, expand onboarding to scaffold them. The framework repo ships only framework files.
2. **Complete #32** — Integration tests proving the compiler detects and recovers from drift scenarios.
3. **Clean release** — Strip maintainer-only artifacts, update README, fix version tags, publish to upstream with release notes.

## Non-Goals

- Changing the onboarding UX beyond adding scaffold steps
- Shipping maintainer specs/plans to the public repo
- Implementing #20 (COLLABORATION.md split) or other backlog items

### Onboarding Expansion

The `/onboard` skill gains a `scaffold-runtime` step that:

1. Creates runtime directories: `.claude/compliance/{compiled,eslint,semgrep,proposals}/`, `.claude/governance/decisions/`, `.claude/governance/guidance/{ceo,cfo,ciso,cko,coo,cto}/`, `.claude/floors/`, `.claude/findings/`, `.claude/metrics/`
2. Copies templates to runtime locations: `templates/compliance/` → `.claude/compliance/`, `templates/governance/` → `.claude/governance/`, `templates/findings/` → `.claude/findings/`
3. Initializes empty runtime files: `.claude/metrics/events.jsonl`
4. Runs initial floor compilation if a floor file exists in `floors/`

Note: The CRO does not have a guidance directory — CRO uses the compliance infrastructure (proposals, change-log) rather than a guidance slot. This is intentional and consistent with the CRO's role as compliance floor guardian.

**Ordering dependency:** The file moves (step 1 of the inventory above) must complete before the onboarding scaffold can reference `templates/compliance/`, `templates/governance/`, and `templates/findings/`. These directories do not exist on current main — the moves create them.

### Verification

Run `/onboard` in a worktree. Verify:

- All runtime paths exist
- Agents, skills, and hooks resolve correctly
- Compiler runs successfully against scaffolded artifacts
- No broken references in CLAUDE.md or agent definitions

## Phase 1b: Drift Detection Testing (#32)

### Test Scenarios

Added to `ops/tests/test-compile-floor.sh`:

| Scenario                                     | Expected Behavior                               |
| -------------------------------------------- | ----------------------------------------------- |
| Clean first compile (no prior artifacts)     | Expected remediation: scaffold and compile      |
| All artifacts present and valid              | No-op, exit 0                                   |
| Artifacts deleted after successful compile   | Unexpected remediation: warn, re-compile        |
| Checksum mismatch (artifact hand-edited)     | Detect drift, warn, re-compile                  |
| Partial compilation (some artifacts missing) | Detect incomplete state, re-compile missing     |
| Floor source changed, artifacts stale        | Detect source-vs-compiled drift, re-compile     |
| `--all` with one floor drifted, one clean    | Re-compiles only the drifted floor              |
| `--verify` flag on all above                 | Detects without writing, exit non-zero on drift |

### Implementation

Tests use fixture floors from `ops/tests/fixtures/`. Each scenario:

1. Sets up a temp directory with controlled initial state
2. Runs `ops/compile-floor.sh` with appropriate flags
3. Asserts on exit code, stdout messages, and file state

## Phase 3: Release Prep

### 3a. Clean Working Tree

- Discard unstaged edits to already-merged specs/plans
- Delete `docs/compliance-coverage.md` (test artifact)

### 3b. Strip Maintainer Artifacts for Upstream Push

Files excluded from the upstream push (kept in private repo only):

- `docs/superpowers/specs/` — design specs
- `docs/superpowers/plans/` — implementation plans
- `memory/` — maintainer context

Method: a release script (`ops/release.sh`) that creates a clean working copy, removes private paths, and pushes to upstream. The private repo retains everything. The script is the single source of truth for what ships and what doesn't.

### 3c. README Refresh

Update `README.md` to reflect the v0.4.0 feature set:

- Multi-floor governance (floors/, CRO, behavioral floor)
- Gomplate-based compliance compiler with schema validation
- Rewards system (kudos/reprimands/behavioral tensions)
- Pre-flight validation with expected vs unexpected remediation
- Progressive examples (5) with enforcement blocks
- 2 new reference guides (COMPILER-GUIDE.md, GOVERNANCE-FLOORS.md)
- Framework purity: clean separation of framework and runtime artifacts

### 3d. Version Correction

On upstream (`rdunie/venutian-antfarm`):

1. Delete tag `v1.1.0`
2. Create tag `v0.3.0` at the same commit (`ef5022c`)
3. Update GitHub release for v0.3.0 with brief note about retag

### 3e. Tag and Release

1. Push v0.4.0 content to upstream main
2. Tag `v0.4.0`
3. Create GitHub release with full release notes covering all shipped features

## Risks

- **Onboarding path changes could break examples**: Mitigated by running examples validation after purity changes.
- **Upstream retag may confuse existing users**: Mitigated by release notes explaining the version correction. Low risk given early adoption stage.
- **Stripped specs/plans create maintenance divergence**: Accepted. The private repo is the source of truth for design history.
